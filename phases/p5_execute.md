# Phase 5：执行吸收循环详细指令

## 目标
按计划书逐项实施，每项完成后验证，持续推进直到用户暂停或全部完成。

## 执行前检查

读取 `absorber-workspace/ABSORPTION_PLAN.md`，找到第一个未完成项（`[ ]`状态）。
确认该项的前置依赖（如有）已完成，否则跳到下一个无依赖的项目。

## 单项执行流程

### 5.1 实施前说明（必须，不可跳过）

在开始任何代码修改前，先输出：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
准备实施 [PA-001]：[特性名称]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
参考来源：[竞品名] 的实现方式
价值：[一句话]
预计改动：
  - [文件/模块1]：[做什么]
  - [文件/模块2]：[做什么]
难度：[低/中/高]

继续？（直接回复即可，或说「跳过」「调整方案」）
```

等待用户确认。用户说任何肯定的话（「好」「继续」「行」「yes」「y」）都算确认。

### 5.2 实施原则

**核心原则：借鉴思路，不照抄代码**
- 参考竞品的设计思路和模式
- 用当前项目的代码风格重新实现
- 保持当前项目的技术栈，不引入不必要的新依赖

**实施步骤：**
1. 读取当前项目相关文件，理解现有实现
2. 读取竞品对应实现（从 `/tmp/absorber/[竞品名]/` 里找）
3. 在当前项目中实施，尽量小步提交
4. 如果实施过程中发现问题（架构冲突、依赖问题），立即停下来说明，不强行继续

**不要做的事：**
- 不要在实施一项的过程中顺手改其他东西
- 不要引入计划外的依赖
- 不要在用户没确认的情况下修改配置文件或关键路径

### 5.3 验证

实施完成后：

**有测试套件的项目：**
```bash
# 运行现有测试，确认没有破坏
[项目的测试命令，从package.json/Makefile/README里找]
```
报告测试结果。如果测试失败，分析原因并修复，不跳过。

**没有测试套件的项目：**
- 人工描述验证方式：「可以通过 [具体操作] 验证该特性已生效」
- 如果是CLI工具，给出具体的命令示例

### 5.4 更新计划书状态

实施完成后，立即更新 `absorber-workspace/ABSORPTION_PLAN.md`：
- 把 `[ ]` 改为 `[x]`
- 在该项下填写实施记录（做了什么，涉及哪些文件）

### 5.5 询问是否继续

```
✅ [PA-001] 完成
[一句话说明实施结果]

进度：X/N 项已完成（P0剩余 Y 项）

下一项：[PA-002] [特性名] | 难度：[低/中/高]
继续？（或说「暂停」「跳到PA-005」）
```

## 异常情况处理

### 实施中发现无法完成
```
⚠️ [PA-001] 实施中止
原因：[具体问题描述]
建议：[降级方案 / 标记为不适用 / 需要用户决策的选项]
```
在计划书中将该项标记为 `[!] 中止 - [原因]`，询问用户如何处理后继续下一项。

### 发现计划书遗漏了重要内容
如果实施过程中发现原计划书遗漏了某个有价值的特性：
- 先完成当前项
- 然后提出新发现，询问用户是否加入计划书
- 不要擅自修改优先级或插队执行

### 用户说「全部自动跑完」
如果用户明确表示要自动执行所有项目，按顺序执行，
但遇到「高难度」项目时仍然要停下来确认，不跳过。

## 循环终止条件

以下任一情况停止循环：
1. 用户说「暂停」「停一下」「先到这」
2. 计划书中所有 P0 项目已完成（停下来汇报，询问是否继续 P1）
3. 所有项目全部完成
4. 遇到无法继续的技术障碍

## 完成汇报格式

所有项目完成后：

```
🎉 吸收完成！

本轮共吸收 N 项特性，来自 M 个竞品。

已完成：
- [特性名1]（来自竞品A）
- [特性名2]（来自竞品B、C）
...

中止项（X项）：
- [特性名]：[原因]

你的项目现在已融合以下竞品的优点：
[竞品列表]

还有 Y 个竞品待分析。继续吸收下一个？
```
