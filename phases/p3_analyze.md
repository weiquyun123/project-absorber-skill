# Phase 3：竞品深度分析详细指令

## 目标
找出竞品有而当前项目没有的、值得吸收的设计亮点。

## 3.1 克隆策略

```bash
# 先尝试浅层克隆（速度快，只要最新快照）
git clone --depth=1 --single-branch [repo_url] /tmp/absorber/[repo_name]

# 如果仓库过大（>500MB）或网络失败，退回到只分析在线文档
# 通过 WebFetch 获取：GitHub页面 + README raw 内容
```

克隆后立即检查大小，超过1GB不克隆，改用在线分析。

## 3.2 分析框架

对每个竞品按以下8个维度扫描，每个维度找「有价值的差异点」：

### 维度1：用户体验与接口设计
- CLI工具：帮助信息质量、命令结构、错误提示友好度、自动补全
- Web应用：路由设计、API一致性、响应格式
- 库/框架：API易用性、默认值合理性、学习曲线

**看哪里**：README的Quick Start、CLI的 `--help` 输出、API文档

### 维度2：配置与扩展机制
- 配置文件格式（YAML/TOML/JSON/env）
- 配置层级（全局/项目/局部）
- 插件/钩子系统

**看哪里**：`config/`、配置文件示例、Plugin相关目录

### 维度3：错误处理与可观测性
- 错误信息质量（是否给出修复建议）
- 日志系统（级别、格式、输出目标）
- 指标/监控集成

**看哪里**：错误处理相关文件、logger模块、`--verbose` 标志实现

### 维度4：测试架构
- 测试覆盖策略（单元/集成/E2E）
- 测试工具选型
- 测试数据管理

**看哪里**：`test/`、`tests/`、CI配置文件

### 维度5：文档与开发者体验
- README结构（是否有示例、截图、动图）
- API文档生成
- CONTRIBUTING.md
- Changelog维护

**看哪里**：根目录文档文件、`docs/`目录

### 维度6：性能与工程质量
- 缓存策略
- 并发处理
- 依赖管理（是否精简）
- 构建/打包优化

**看哪里**：核心模块、package.json/Cargo.toml依赖列表

### 维度7：社区与生态
- Issue模板
- PR模板
- 社区集成（Discord/Slack链接、Badge）

**看哪里**：`.github/`目录

### 维度8：安全性
- 输入验证
- 敏感信息处理（secrets、API keys）
- 权限控制

**看哪里**：输入处理代码、认证相关模块

## 3.3 优先级评分规则

| 分级 | 定义 | 例子 |
|------|------|------|
| P0 | 当前项目明显缺失，用户会直接感知到差距 | 无错误提示 / 无配置文件支持 |
| P1 | 有价值的改进，不是必须但会明显提升质量 | 更好的日志格式 / 插件机制 |
| P2 | 锦上添花，有精力再做 | 更多badge / changelog自动生成 |

## 3.4 不吸收的判断标准

遇到以下情况标记为「不吸收」并说明原因：
- **架构冲突**：需要重写核心才能支持
- **依赖过重**：引入庞大依赖只为一个小特性
- **场景不符**：竞品面向不同使用场景
- **已被覆盖**：当前项目有等效实现
- **技术债**：竞品的该特性实现质量差，不值得借鉴
