# Phase 2：竞品发现详细指令

## 目标
找到 5-10 个真实存在、质量靠谱的同类开源项目，建立吸收队列。

## 搜索策略

### 第一轮：GitHub直接搜索
基于 Phase 1 生成的关键词，构造以下搜索查询（WebSearch）：

```
site:github.com [功能关键词] stars:>500
site:github.com awesome-[领域词]（找curated list）
[技术栈] [项目类型] open source github
```

每组关键词搜索后，从结果里提取GitHub仓库链接。

### 第二轮：Awesome列表挖掘
很多领域有 `awesome-xxx` 汇总列表，里面已经筛选过质量。
用 WebFetch 拉取相关 awesome 列表页面，提取项目链接。

### 第三轮：交叉验证
从已找到的竞品README里，看它们提到哪些「同类项目」或「对比项目」，
这些往往是该领域最被认可的竞品。

## 筛选标准

**必须满足（否则跳过）：**
- GitHub Stars ≥ 500（说明有一定用户基础）
- 最近6个月内有提交（非死项目）
- 主要语言与当前项目相同或兼容（能借鉴代码模式）
- 方向与当前项目重叠度 ≥ 50%

**加分项（排序参考）：**
- Stars 越多越靠前
- 有活跃的 Issue 和 PR 讨论（说明功能演化活跃）
- 有详细文档或专门的 docs 站
- 被知名项目引用或推荐

**排除：**
- Fork 数量异常高但 Stars 低（可能是课程作业）
- README 只有几行、明显不完整
- 最后提交超过1年
- 与当前项目方向差异过大（避免噪声）

## 输出格式

整理候选清单，存入 `absorber-workspace/competitor_list.md`：

```markdown
# 竞品候选清单

生成时间：[日期]
搜索关键词：[使用的关键词]

## 推荐吸收顺序

| 排名 | 项目 | Stars | 最后更新 | 相关性理由 |
|------|------|-------|----------|------------|
| 1 | [名称](URL) | 12.3k | 2024-01 | [一句话说明] |
| 2 | ... | | | |

## 排除项目

| 项目 | 排除原因 |
|------|----------|
| [名称] | 最后提交2022年 |
```

## 呈现给用户确认

```
✅ 发现 N 个竞品候选，按相关性排序：

1. **[项目名]**（⭐ X.Xk）
   [一句话描述为何值得分析]
   
2. **[项目名]**（⭐ X.Xk）
   ...

建议按此顺序逐个吸收。是否确认？
可以：
- 直接说「确认」开始分析
- 说「去掉第X个」调整列表
- 说「加上 [项目URL]」手动添加
```

等待用户确认后，保存最终列表并进入 Phase 3。

## 边界情况处理

**如果搜索结果很少（<3个）：**
- 放宽 Stars 门槛至 100
- 扩大关键词范围（用更通用的领域词）
- 告知用户并说明已调整策略

**如果项目方向太小众，找不到同类：**
- 搜索「上位概念」（如：找不到同类CLI工具，改搜同语言的优秀CLI工具）
- 明确告知用户：「该方向开源项目较少，建议参考以下相邻领域的项目」

**如果用户已知道要分析哪些竞品：**
- 用户可直接提供URL列表，跳过搜索，直接进入 Phase 3
